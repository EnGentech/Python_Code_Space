Project Overview

Scroll to Table of Contents

In this capstone project, you will collaborate with your peers in groups of approximately three members to develop a Python application that processes financial data. The application will receive transaction data from a CSV file, process it by performing various operations such as updating account summaries and identifying suspicious transactions, logs the processing events to a log file, and output the processed data to another file (CSV).

Each of you will be responsible for one of the project's three main components:

  • Input Handling: Reading and validating the input data from a CSV file, and implementing custom data validation rules.
  • Data Processing: Processing the input data, updating account summaries, checking for suspicious transactions, and maintaining transaction statistics, while also implementing a configurable logging mechanism.
  • Output Handling: Exporting the processed data to appropriate output files, with options for custom filtering.

You will be assigned specific tasks via GitHub issues and will be responsible for resolving these issues to complete your portion of the project. The project will be developed in a single repository using Git and GitHub, following established software development practices such as creating branches, submitting pull requests, and performing code reviews.

Throughout the project, you will learn and practice various skills, including Python programming, working with CSV and JSON files, implementing data validation and processing algorithms, using Git and GitHub for collaborative development, and adhering to group work guidelines to ensure successful project completion.

Design Requirements

Scroll to Table of Contents

Please follow these requirements when completing your portion of the project:

Data Input Format:

The input data files should be in one of the following formats:

  • CSV (Comma-Separated Values)
  • JSON (JavaScript Object Notation)

The input data should contain the following fields:

  • Transaction ID
  • Account number
  • Date
  • Transaction type
  • Amount
  • Currency
  • Description

Data Output Format:

The output data files should be in one of the following formats:

  • CSV (Comma-Separated Values) for account summaries and suspicious transactions

Account summaries output should contain the following fields:

  • Account number
  • Balance
  • Total Deposits
  • Total Withdrawals

Suspicious transactions output should contain the following fields:

  • Transaction ID
  • Account number
  • Date
  • Transaction type
  • Amount
  • Currency
  • Description

Transaction statistics output should contain the following fields:

  • Transaction type
  • Total amount
  • Transaction count

Log File Format:

The log file format should be plain text, with each log entry containing the following information:

  • Timestamp
  • Log level (e.g., INFO, WARNING, ERROR, etc.)
  • Log message

Example:


2023-06-02 15:39:50,765 - INFO - Account summary updated: 1001

The log file should record important events, warnings, and errors that occur during the processing of financial data. Users should be able to configure the log level and log output format.

Expected Output:

The application should process the input data and generate the following outputs:

  • Account summaries for each account, including the account balance, total deposits, and total withdrawals
  • A list of suspicious transactions based on predefined or custom criteria
  • Transaction statistics, including the total amount and transaction count per transaction type
  • A log file containing logs of important events, warnings and errors.

Dealing with Git Conflicts

Scroll to top of Guides for Students

When working with a shared repository, it's common to encounter conflicts between different code versions. Students should review the following guides to learn how to deal with Git conflicts:

  • GitHub Docs: Resolving a Merge Conflict
  • Atlassian: How to Resolve Conflicts in Git

Conflicts Guide

  1. Identify the conflict: First, you'll receive a notification that there's a conflict when you try to merge branches or pull changes. Git will tell you which files have conflicts. In this project, since each of you will be working within a specific file for your assigned class, the most likely file to have conflicts is main.py.
  2. Pull the latest changes: Before resolving the conflict, ensure you have the latest changes from the remote repository. Run git pull origin main to fetch the latest updates.
  3. Open the conflicted file: Open the main.py file in your favourite text editor or IDE. Git will mark the conflicting sections with special markers (<<<<<<<, =======, and >>>>>>>). These markers separate the conflicting changes made by you and the other contributor.
  4. Review the changes: Carefully review the changes between the markers. The changes made by you will be between <<<<<<< and =======, while the changes made by the other contributor will be between ======= and >>>>>>>.
  5. Resolve the conflict: Decide which changes should be kept, or if a combination of the changes is appropriate. Edit the file to remove the markers and keep only the desired changes. Make sure the code remains functional and adheres to the project requirements.
  6. Test the changes: Before committing the resolved conflict, test the code to ensure everything works as expected. Run the application and verify that there are no issues or errors.
  7. Commit and push the changes: Once you're satisfied with the resolution, stage the changes with git add main.py, and then commit them using git commit -m "Resolved conflict in main.py". Finally, push the changes to the remote repository with git push.

Code Reviews

Scroll to top of Guides for Students
  1. Understand the changes: Begin by understanding the purpose of the changes in the pull request. Read the issue description and any comments provided by the author to understand the context and the expected outcome.
  2. Check the code: Go through the code changes in the pull request. Look for the following:
    • Are the changes addressing the issue?
    • Is the code easy to understand, or are there any confusing or unclear parts?

Ethical Practices for Using Online Resources

Scroll to top of Guides for Students
  • Understand the Code: Before using any code found online, ensure you understand how it works and how it relates to your specific assignment. If you do not understand the code, you may not be able to adapt it properly or use it effectively in your project.
  • Give Credit to Original Authors: Always give credit to the original authors of the code you use in your assignment. This is not only an ethical practice but also helps to promote a culture of collaboration and respect among developers. You can give credit by citing the source in your code comments.
  • Proper Citation Styles: When citing code sources in your comments, use a consistent citation style. Here are some examples:
    • For code snippets found on websites, you can use the following format: 
      
# Source: [Author Name (if available)], [Title of the webpage], [URL], [Date accessed]
    • For code snippets from forums (e.g., Stack Overflow), you can use the following format: 
      
# Source: [Author Name (if available)], [Title of the forum thread], [URL], [Date accessed]
# Answer by: [Answer author name (if available)], [Answer author's profile URL (if available)]
    • For code snippets from GitHub repositories or other version control systems, you can use the following format: 
      
# Source: [Author Name (if available)], [Repository Name], [URL], [Commit ID (if available)], [Date accessed]
  • Adapt and Customize the Code: Adapt and customize the code you find online to fit the specific requirements of your assignment. Avoid copying and pasting code verbatim without making any modifications. This demonstrates that you understand the code and can apply it to solve your unique problem.
  • Avoid Plagiarism: It is important to remember that using code from online sources should be a tool for learning and should not be used as a way to avoid writing your own code. Make sure to always contribute your own ideas and efforts to your assignments.
  • Respect Licenses: Always check and respect the licenses associated with the code you use. Some code may have specific requirements or restrictions regarding its use, modification, or distribution. Make sure you comply with these terms when using the code in your assignment.

To make the most of the limited time available for face-to-face reviews with instructors, it's important to come prepared. Follow these guidelines to ensure an efficient and productive review session:
